# PO Numbering — Backend Specification

Status: **Backend change required.** The frontend implements the canonical formatter
(`src/utils/poNumber.ts`) for previews, validation, and display, but the authoritative
PO number (specifically the per-supplier-per-day serial) **must** be generated by the
backend at PO generation time.

## 1. Required format

```
PO-DDMMYY-SupplierToken-NNNN
```

| Segment | Rule | Example |
|---------|------|---------|
| `PO` | Literal prefix | `PO` |
| `DDMMYY` | PO creation date, 2-digit day + month + year, **no separators** | `220626` (22 Jun 2026) |
| `SupplierToken` | Supplier name with all non-alphanumeric characters removed, original casing preserved | `MochenzComputers`, `AlFatahTrading` |
| `NNNN` | 4-digit zero-padded serial, **resets per supplier per day** | `0001` |

Full example: a PO created on 22 Jun 2026 for **Mochenz Computers**, first of that day,
is `PO-220626-MochenzComputers-0001`; the second is `PO-220626-MochenzComputers-0002`.

## 2. Supplier token algorithm (must match frontend `normalizeSupplierToken`)

1. Take the full supplier name.
2. Remove every non-alphanumeric character (spaces, punctuation, symbols) — this
   concatenates all words together (so `Al-Fatah Trading` → `AlFatahTrading`).
3. Preserve original casing. Cap at 30 characters.
4. If nothing remains, use `Vendor`.

Examples:
- `Mochenz Computers` → `MochenzComputers`
- `Al-Fatah Trading` → `AlFatahTrading`
- `3M Nigeria Ltd` → `3MNigeriaLtd`
- `  &Co Supplies` → `CoSupplies`

## 3. Serial algorithm

- The serial is scoped to **(supplier_token, creation_date)**.
- It increments atomically; the first PO for a supplier on a given calendar day is `0001`.
- Use the **normalised supplier token** (not the raw vendor id) as the scope key so that
  the human-readable serial matches what appears in the PO number. (Optionally also key on
  `vendor_id` if you want to disambiguate two different vendors that normalise to the same
  token — but the displayed serial must still read correctly per token per day.)
- Generation must be race-safe (DB unique constraint or row lock / sequence per
  `(token, date)`), returning a fresh serial under concurrent generation.

Recommended uniqueness guard: a unique index on the final `po_number` string, plus a
retry on collision.

## 4. Where to apply

- `POST /api/mrfs/{id}/generate-po` (finalise, not `save_as_draft`) must assign and persist
  `po_number` in the new format, and return it on `data.mrf.po_number`.
- Draft saves (`save_as_draft: true`) should **not** burn a serial; assign the real number
  only on finalise. If a number was reserved for a draft, ensure discarding the draft frees it.
- Regeneration (`regenerate: true`) of an existing PO should **keep the existing
  `po_number`** (do not allocate a new serial) unless product decides otherwise.
- `POST /trips/{tripId}/generate-trip-po` (logistics trip PO) should follow the same format,
  using the carrier/vendor name as the supplier token. (Today this number is user-entered;
  ideally the backend generates it the same way.)

## 5. Backward compatibility

- **Existing PO numbers must not be changed.** Only POs generated after deployment use the
  new format.
- All display surfaces already read `po_number || poNumber` verbatim, so existing numbers
  continue to render as-is and new numbers render in the new format automatically.

## 6. Frontend support already in place

- `src/utils/poNumber.ts`
  - `formatPoDatePart`, `normalizeSupplierToken`, `formatPoSerial`
  - `buildPoNumber`, `parsePoNumber`, `isNewFormatPoNumber`
  - `previewPoNumber(supplierName, date)` → `PO-220626-MochenzComputers-XXXX` placeholder for pre-finalise UI
- The PO create form preview and the legacy client-supplied path use these helpers so any
  client-side number matches the backend format.

## 7. Acceptance criteria

- [ ] New PO for Mochenz Computers on 22 Jun 2026 → `PO-220626-MochenzComputers-0001`.
- [ ] Second new PO same supplier/day → `PO-220626-MochenzComputers-0002`.
- [ ] New PO for Al-Fatah Trading → `PO-220626-AlFatahTrading-0001` (independent serial).
- [ ] Next calendar day, serial resets to `0001` per supplier.
- [ ] Concurrent generations never produce duplicate `po_number`.
- [ ] Existing POs retain their original numbers.
- [ ] `po_number` returned on `generate-po` finalise response and on `GET /api/mrfs` rows.
